# バトル要素 設計仕様（学習統合型）

下の子からの「バトルしたい」という要望を、**学習アプリの目的を損なわず**に取り込むための仕様。基本思想は「**攻撃＝学習タスク**」。バトルの操作そのものが問題を解くことになるため、遊ぶ＝学ぶになり distraction を最小化する。

## 1. 目的と非目標

**目的**
- 学習の動機付けを強める（捕獲・習得した虫が「強さ」になる）
- 新しい学びを少量織り込む（昆虫の分類・生態という理科の入口）

**非目標（やらないこと）**
- リアルタイム/アクション/反射神経の要素
- PvP・チャット・対人の競争圧
- 問題を解かずに進む独立ミニゲーム、スタット稼ぎ周回

## 2. 設計原則

1. **攻撃は必ず1問**。スキル不要の操作で進む要素を入れない
2. **セッションモード化**し、学習としてカウントする（ミッション／ふくしゅうと並列）
3. **時間制限**（1戦 5〜8問）で無限ループを防ぐ
4. **報酬は既存経済（琥珀・捕獲）と同等**にしてfarmingを防ぐ
5. **低ストレス・PvPなし・煽りなし**。敗北ペナルティは軽微（学習記録は必ず残す）

## 3. コア仕様（A：クイズ駆動ターンバトル）— 確定方針

### 3.0 バトルは「練習のごほうび」＝琥珀型ゲート（最重要・anti-distraction）
「子どもがバトルばかりになる」リスクを**構造で**防ぐ。バトルは自由に遊べず、**通常練習で解禁**する：
- 通常問題の正解で**バトルゲージ**が溜まる（採集ゲージと同型）。満タン（例：20〜30問正解）で**ボス挑戦が1回解禁**
- **ランダム戦は作らず、ボス戦のみ**。これにより頻度が自然に制限される
- バトル中も問題を解く（§3.1）ので、**練習で解禁＋戦闘中も解答**の二重ゲートになる

### 3.1 戦闘ループ（HP制・攻防とも1問）
- 構成：**自分のパーティ（捕獲した虫） vs ボス**、HP制、ターン制。HPは双方5から開始
- **こうげきターン**：1問 → 正解で相手HP−1、不正解はダメージなし（テンポを失うだけ）
- **ぼうぎょターン**：1問 → 正解でガード（無傷）、不正解で自分HP−1
- **HPの増減は必ず1問に紐づく**。先にHP0になった方が負け（おおよそ5〜10問で決着）
- CPUはサイコロを振らない（HP増減は全て子どもの解答で決まる「HPの的」）。勝つ＝ちゃんと答えられた、になり学習と一致
- 出題は**既存のSRS/カリキュラムから**（計算＝計算／漢字＝スキル／英単語＝各形式）。新しい問題タイプは不要
- 勝敗に関わらず、解いた問題は**学習として記録**（streak・習得カウント・SRSに反映）

### 3.2 パーティ編成（コレクション価値）／技は一律にする（balance safe）
- **技の強弱・技の内容に差をつけない**（凝ると破綻するため）。こうげき成功＝**一律ダメージ1**
- 虫の違いは **HP（レア度）と相性タイプだけ**：
  - HP例：N/R=1, SR/SSR=2, SS=3
  - パーティ（例3匹）を編成 → **HP合計＝粘り強さ**。倒れたら次の虫が出る。**集める・レアを揃えるほど長く戦える**＝コレクションが直接「強さ」になる（多段スタット・技ツリー不要）

### 3.3 年齢差の吸収
- 上の子：攻撃も防御も問題（歯ごたえ）
- 下の子(5歳)：**防御は自動**にして攻撃問題のみ・HP小さめ（やさしいモード）
- 難度トグル1つで両立。CPU AIの作り込み不要

## 4. 学習レイヤー（B：分類による相性 ＝ 新しい理科の学び）

- 虫データには既に `order` / `family` がある。これを**タイプ**に写像（例：甲虫／チョウ・ガ／カマキリ／ハチ／トンボ …）
- **相性**をシンプルなじゃんけん的関係で導入（例：カマキリ＞チョウ＞アブラムシ的な…具体表は§8で要決定）
- 効果：
  - 相性有利な虫で挑むとダメージ増 → 「**どの虫で挑むか**」の選択が生まれる＝分類の学び
  - 相性発動時に**一言の生態知識**を表示（例「カマキリは チョウを たべる」）＝ incidental science learning
- 「どの虫を出すか」は図鑑の捕獲状況と連動（持っている虫からチーム編成）

## 5. 強さの算出（習得を動機化）

- §3.2 のとおり、強さは**パーティのHP合計**で表す（レア度→HP、頭数→粘り）。倍率や複雑なスタットは使わない
- 相性（§4）は入れる場合のみ**有利でダメージ＋1**程度（じゃんけん相当）。表1枚で済む範囲に留める
- 結果として「強くなる＝もっと集める／レアを揃える」が成立し、バトルが**コレクションと学習の動機**になる

## 6. ボス戦（C：習得チェックポイントの再フレーム）

- N個マスターで「ボス昆虫」が解放
- 撃破には**連続正解（＝ミニテスト相当）**が必要 → 復習・テストをボス戦として再フレーム
- 撃破報酬：バッジ＋たまにレア確定。ただし**クリア済みボスの周回はディスカウント**（漢字チャレンジの周回封じと同じ方針で farming を防ぐ）

## 7. distraction を防ぐガードレール（チェックリスト）

- [ ] すべての「こうげき」が1問になっているか
- [ ] バトルが通常練習を置き換えていないか（セッションとしてカウント／報酬同等）
- [ ] 1戦が短く時間制限されているか
- [ ] PvP・リアルタイム・煽り表現がないか
- [ ] 敗北が過度なストレスにならないか（やさしいフィードバック）

## 8. 未決事項（要ユーザー判断）

- **相性表の具体**：どのタイプがどれに有利か（生態的に正しく、かつ子どもに分かりやすい関係に）
- **3ゲーム共通か個別か**：バトルを各ゲーム内モードにするか、ポータルに横断バトルを置くか
- **報酬の重み**：勝利報酬・ボス報酬の琥珀/捕獲量（既存経済とのバランス）
- **対象年齢**：下の子（5歳）向けの簡易UIと、上の子向けの歯ごたえをどう両立するか
- **CPUの強さ**：固定難度か、習得度に応じた自動調整か

## 9. 実装の見取り図

- 既存の出題エンジン（各ゲームのquiz/SRSループ）を**再利用**し、バトル用の「HP・ターン・演出」ラッパを被せる
- 虫データに `battleType`（order/family から導出）を付与（マッピング1枚。`shared/bugs.js` か派生で）
- `shared/battle.js`（共通ロジック：HP計算・相性・ターン進行）＋各ゲームのUI薄層
- 報酬は既存 `Q4BReward` 経済に統合（勝利/ボスの付与をそこに通す）

## 10. フェーズ実装計画（独立出荷可能）

## 11. まとめ

A（クイズ駆動ターンバトル）を**セッションモードの追加**として実装するのが最も安全で、学習フローを壊さずに動機付けと「分類・生態」という新しい学びを足せる。リアルタイム/PvP/スタット稼ぎ系は採用しない。
